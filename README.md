### 3. **Liskov Substitution Principle (LSP) - Principio de Sustituci√≥n de Liskov**

Este principio dice que los objetos de una clase derivada deben ser sustituibles por objetos de la clase base sin alterar la funcionalidad del programa. En otras palabras, las subclases deben ser completamente intercambiables con sus clases base.

Ejemplo:

```kotlin

    open fun arrancar() = println("Veh√≠culo arrancando")

        throw Exception("Las bicicletas no arrancan")
    }
}

// Sigue LSP

    open fun arrancar() = println("Veh√≠culo arrancando")

class Coche : Vehiculo() {
    override fun arrancar() = println("Coche arrancando")
}
```

### 4. **Interface Segregation Principle (ISP) - Principio de Segregaci√≥n de Interfaces**

Este principio sugiere que una clase no debe ser forzada a implementar interfaces que no necesita. Es mejor tener interfaces espec√≠ficas para cada tipo de comportamiento que una √∫nica interfaz general.

Ejemplo:

```kotlin
// No sigue ISP

    fun imprimir()
    fun escanear()
    fun faxear()
}

    override fun imprimir() {
        println("Imprimiendo")
    }

    override fun escanear() {
        throw Exception("No escanea")
    }

    override fun faxear() {
        throw Exception("No faxea")
    }
}

// Sigue ISP

    fun imprimir()
}

interface Escaner {
    fun escanear()
}
```

### 5. **Dependency Inversion Principle (DIP) - Principio de Inversi√≥n de Dependencias**

Las clases de alto nivel no deben depender de clases de bajo nivel. Ambas deben depender de abstracciones (interfaces o clases abstractas). Adem√°s, las abstracciones no deben depender de los detalles, los detalles deben depender de las abstracciones.

Ejemplo:

```kotlin
// No sigue DIP
class Motor {
    fun arrancar() {
        println("Motor arrancando")
    }
}

class Coche {
    private val motor = Motor()
    
    fun arrancar() {
        motor.arrancar()
    }
}

// Sigue DIP
interface Arrancable {
    fun arrancar()
}

class Motor : Arrancable {
    override fun arrancar() {
        println("Motor arrancando")
    }
}

class Coche(private val motor: Arrancable) {
    fun arrancar() {
        motor.arrancar()
    }
}
```

Estos principios SOLID ayudan a crear sistemas de software m√°s modulares, f√°ciles de extender y mantener, promoviendo un dise√±o robusto y flexible.

### üîπ **Tipos de Herencia en POO**

1Ô∏è‚É£ **Especificaci√≥n** ‚Üí La clase hija es una versi√≥n m√°s concreta de la clase padre.

- Ejemplo: `Vehiculo ‚Üí Coche` (Coche hereda de Vehiculo y a√±ade m√°s detalles).

2Ô∏è‚É£ **Especializaci√≥n** ‚Üí Se modifica o mejora un comportamiento de la clase padre.

- Ejemplo: `Empleado ‚Üí Gerente` (Gerente redefine c√≥mo se calcula el bono).

3Ô∏è‚É£ **Extensi√≥n** ‚Üí Se agregan nuevas funcionalidades sin cambiar la base.

- Ejemplo: `Animal ‚Üí Perro` (Perro hereda respirar() pero a√±ade ladrar()).

4Ô∏è‚É£ **Construcci√≥n** ‚Üí Se define una estructura base que las clases hijas deben seguir.

- Ejemplo: `Figura ‚Üí C√≠rculo/Rect√°ngulo` (todas deben implementar calcularArea()).

üîπ **Resumen:**

- **Especificaci√≥n** ‚Üí Hacer m√°s concreta la clase.
- **Especializaci√≥n** ‚Üí Modificar el comportamiento.
- **Extensi√≥n** ‚Üí Agregar funcionalidades.
- **Construcci√≥n** ‚Üí Establecer una estructura obligatoria.
